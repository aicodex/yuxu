# 一、Agent 迭代方法（机器侧）

## 1. Reflexion（失败驱动自我反思）

**来源**：Shinn et al., NeurIPS 2023

**核心思想**：用语言反馈替代梯度，让 Agent 通过"自我反思"从错误中学习

**适用场景**：
- 任务执行失败后（代码报错、答案错误、任务超时）
- 需要分析"错在哪、为什么、怎么改"的场景
- 不需要外部奖励信号，纯靠 LLM 自己判断

**不适用**：
- 任务成功且结果满意时
- 没有失败轨迹可供分析时

**步骤**：
```
1. 记录失败轨迹（question + thought/action/observation）
2. 用 LLM 分析："哪里错了、为什么、怎么改"
3. 反思结果存入 list（最多3条）
4. 下次类似任务，prompt 注入这些反思
5. 重复直到成功
```

**触发条件**：`is_correct() == False` 或超过 max_steps

---

## 2. Voyager（技能库迭代）

**来源**：Minecraft Agent, 2023

**核心思想**：把成功执行的代码固化为技能，存入技能库，下次复用

**适用场景**：
- 重复性任务（做网页、读论文、写代码）
- 任务可以分解为可执行步骤
- 需要跨任务复用经验

**不适用**：
- 一次性任务（不需要复用）
- 纯推理类任务（没有可固化的代码）

**步骤**：
```
1. 用 embedding 检索相关已有技能
2. 生成代码 → 执行 → 获取反馈
3. 反馈注入下一轮，迭代改进（最多4轮）
4. 成功后生成技能描述
5. 向量化描述 + 代码存入技能库
```

**触发条件**：代码执行失败 / 4轮后仍无法完成

---

## 3. STaR（自举推理学习）

**来源**：Self-Taught Reasoner, 2023

**核心思想**：用模型自己生成的成功推理链微调自己

**适用场景**：
- 有标准答案的任务（数学、推理题）
- 可以生成大量推理过程
- 愿意花时间做多轮迭代训练

**不适用**：
- 没有标准答案的开放任务
- 不能做模型微调的场景

**步骤**：
```
1. 生成推理过程（不管对错）
2. 正确答案 → 直接作为正例
3. 错误答案 → 告诉正确答案，重新生成推理
4. 用筛选后的样本微调模型
5. 下一轮用微调后的模型重新生成
```

**关键洞察**：错误样本"逆向合理化"后再学

---

## 4. SPIN（自我博弈）

**来源**：Self-Play Fine-Tuning, 2024

**核心思想**：让模型自己区分"自己的输出"和"人类写的好的输出"

**适用场景**：
- 有高质量人类标注数据的任务
- 希望模型自己学会区分好坏
- 可以做多轮迭代训练

**不适用**：
- 没有可靠的人类标注数据
- 不能做模型训练的部署场景

**步骤**：
```
1. 当前模型生成"对手样本"
2. 训练模型给"人类样本"高分、"自己生成的"低分
3. 训练后的模型成为新"对手"
4. 循环直到分不清两者区别
```

---

## 5. AgentFactory（北大）

**来源**：北京大学, 2026

**核心思想**：把成功解法固化成 Python 子 agent + agent.md

**适用场景**：
- 复杂任务需要分解为多个子 agent
- 希望 agent 可以导出到其他框架（LangChain、Claude Code）
- 需要创建可复用的技能模块

**不适用**：
- 简单一次性任务
- 不需要跨框架复用的场景

**步骤**：
```
1. Meta-Agent 分解任务，创建 subagent（代码 + agent.md）
2. 执行成功 → 保存入库
3. 执行失败/质量下降 → 检索已有 subagent，修改使其更健壮
4. 成熟的 subagent 可导出
```

---

## 6. Self-Refine（生成→反馈→改进 循环）

**来源**：Reflexion 的简化版精神继承

**核心思想**：重复"生成 → 评审反馈 → 改进"直到满意

**适用场景**：
- 生成类任务（写代码、写文章、做设计）
- 有明确的质量标准可以自评
- 不需要外部环境反馈

**不适用**：
- 环境反馈至关重要的任务
- 无法自评质量的任务

**步骤**：
```
1. 生成初始响应
2. LLM 评审给出具体改进意见
3. 反馈注入下一轮 prompt
4. 重复直到质量达标
```

# 二、人类反思方法论（方法侧）

## 管理学方法论

### 1. PDCA 循环

**来源**：戴明（Edwards Deming）推广，又称"戴明环"

**适用场景**：
- 有明确目标的重复性任务
- 需要持续改进的流程
- 项目管理、个人习惯养成

**不适用**：
- 目标模糊的探索性任务
- 一次性任务

**步骤**：
```
P(Plan) → D(Do) → C(Check) → A(Act) → 循环
```

**触发问题**："这次计划执行得怎么样？哪些对了/错了？"

---

### 2. AAR（After Action Review）事后回顾

**来源**：美国陆军《领导者复盘指南》

**适用场景**：
- 项目/任务完成后
- 事件结束后 24-48 小时内
- 团队复盘会议

**不适用**：
- 任务进行中（应该用 PDCA）
- 没有具体事件可供回顾

**6问**：
```
1. 当初的意图是什么？
2. 实际发生了什么？
3. 怎么发生的？
4. 为什么会发生？
5. 从中学到了什么？
6. 接下来可以做什么？
```

---

### 3. KPT 复盘法

**来源**：日本企业

**适用场景**：
- 小团队或个人日常快速复盘
- 轻量级，不需要深入分析
- 频繁的短周期迭代

**不适用**：
- 重大项目需要深入复盘
- 需要记录详细原因分析

**3项**：
```
Keep —— 哪些有效？
Problem —— 遇到什么问题？
Try —— 下次尝试什么？
```

---

### 4. 5Why 分析法

**来源**：丰田汽车大野耐一

**适用场景**：
- 出了问题要找根本原因
- 避免治标不治本
- 质量分析、故障诊断

**不适用**：
- 没有明显问题要分析
- 需要快速做决策时

**步骤**：连续问"为什么"，直到找到根本原因

**触发问题**："这是根本原因还是症状？"

---

### 5. SWOT 分析

**来源**：旧金山大学管理学教授，1980 年代

**适用场景**：
- 战略规划
- 竞品分析
- 项目启动前的全面评估

**不适用**：
- 具体执行问题
- 需要快速决策时

**4 象限**：Strengths / Weaknesses / Opportunities / Threats

---

### 6. MECE 分析法

**来源**：麦肯锡芭芭拉·明托《金字塔原理》

**适用场景**：
- 问题分解
- 分类整理
- 避免遗漏和重叠

**不适用**：
- 开放式探索
- 不需要分类的简单问题

**原则**：相互独立、完全穷尽

---

### 7. 3W 法则

**来源**：管理学基础框架

**适用场景**：
- 任务开始前澄清需求
- 接受新任务时
- 确保理解正确

**不适用**：
- 执行过程中的调整
- 复盘分析

**3 问**：
```
What —— 是什么？要做什么？
Why —— 为什么做？不做会怎样？
How —— 怎么做？
```

---

### 8. OKR

**来源**：Intel → Google 推广

**适用场景**：
- 团队目标对齐
- 季度/年度规划
- 需要聚焦重要的事

**不适用**：
- 日常具体执行
- 个人任务管理（太重）

**步骤**：O（目标）→ KR（关键结果）→ 评分 → 对齐

---

### 9. 敏捷回顾会（Sprint Retrospective）

**来源**：敏捷软件开发

**适用场景**：
- 团队迭代后
- 固定仪式化复盘
- 需要团队共同参与

**不适用**：
- 个人任务
- 不需要团队参与的任务

**5 步**：开场 → 收集数据 → 产生见解 → 决定改进项 → 结束

---

## 哲学/科学方法论

### 10. 苏格拉底提问法

**来源**：古希腊苏格拉底

**适用场景**：
- 重大决策前
- 澄清概念
- 检验假设

**不适用**：
- 需要快速做决策时
- 日常执行任务

**6 步追问**：
```
1. 澄清命题 —— X 是什么意思？
2. 澄清假设 —— 结论背后的假设是什么？
3. 验证推理 —— 如何从 A 推到 B？
4. 探索反例 —— 有没有不同视角？
5. 探索后果 —— 会导致什么？
6. 回归原命题 —— 最初论点还成立吗？
```

---

### 11. 科学方法论

**来源**：培根《新工具》

**适用场景**：
- 研究设计
- 实验验证
- 假设检验

**不适用**：
- 日常问题
- 不需要验证的判断

**步骤**：观察 → 假说 → 预测 → 实验 → 分析 → 结论

---

### 12. 辩证法（正反合）

**来源**：黑格尔

**适用场景**：
- 战略决策
- 理论构建
- 价值观梳理

**不适用**：
- 具体执行问题
- 需要快速结论

**3 步**：正题 → 反题（找矛盾）→ 合题（综合）

---

### 13. 奥卡姆剃刀

**来源**：威廉·奥卡姆

**适用场景**：
- 有多个等效解释时
- 防止过度复杂化
- 选择最简单的解决方案

**触发问题**："有没有更简单的解释？"

---

### 14. 批判性思维 8 问

**来源**：西方哲学和教育学

**适用场景**：
- 评估方案
- 审阅他人工作
- 避免被表面逻辑说服

**不适用**：
- 头脑风暴阶段
- 需要快速产出

**8 问**：
```
1. 论题、结论、理由是什么？
2. 有无不明确的词语？
3. 假设是什么？
4. 有无谬误？
5. 证据效力如何？
6. 有无替代原因？
7. 数据有无欺骗性？
8. 重要信息被忽略？
```

---

### 15. 决策树分析

**来源**：运筹学

**适用场景**：
- 不确定性下的选择
- 需要量化比较各选项
- 重大投资/方向决策

**不适用**：
- 信息完全透明的情况
- 情感主导的决策

---

### 16. 费米估算

**来源**：物理学家费米

**适用场景**：
- 信息不全时快速估算
- 验证某个数量级是否合理
- 面试/快速分析

**触发问题**："这个数字合理吗？有办法验证吗？"

# 三、场景-方法路由表

## 按任务类型

| 任务类型 | 推荐方法 |
|----------|----------|
| 重复性执行任务（写代码、做网页） | Voyager 技能库 + PDCA |
| 有标准答案的推理任务 | STaR / SPIN |
| 开放性生成任务 | Self-Refine + AAR |
| 复杂任务分解 | AgentFactory + MECE |
| 失败后分析 | 5Why + Reflexion |

## 按迭代触发条件

| 触发条件 | 应该用什么 |
|----------|-----------|
| 任务失败 | 5Why → Reflexion |
| 任务完成但效果不好 | AAR / KPT → PDCA |
| 开始新任务目标模糊 | 3W → 苏格拉底提问 |
| 持续改进流程 | PDCA 循环 |
| 团队迭代复盘 | 敏捷回顾会 |
| 战略规划/竞品分析 | SWOT + MECE |
| 验证假设/研究 | 科学方法论 |
| 评估论证/防忽悠 | 批判性思维 8 问 |
| 重大决策 | 苏格拉底提问 + 决策树 |

## 按 Agent 类型

| Agent 类型 | 适合的迭代方法 |
|------------|---------------|
| 代码执行类 Agent | Voyager 技能库 + Reflexion |
| 研究分析类 Agent | Self-Refine + AAR + 科学方法论 |
| 任务规划类 Agent | PDCA + OKR + SWOT |
| 对话/客服类 Agent | SPIN + 批判性思维 |
| 多 Agent 协作 | AgentFactory + 敏捷回顾 |

# 四、实用组合推荐

## 组合 1：代码 Agent

```
任务执行
    ↓
失败？ → 5Why + Reflexion（分析错哪了）
    ↓
成功后 → Voyager 风格（固化成功代码到技能库）
    ↓
定期 → PDCA 循环（持续优化流程）
    ↓
有生成物 → Self-Refine（迭代改进直到满意）
```

## 组合 2：研究 Agent

```
任务执行
    ↓
完成后 → AAR 6 问（复盘学到了什么）
    ↓
发现新问题 → 科学方法论（提出假说 → 验证）
    ↓
重大决策 → 苏格拉底提问（检验假设）
    ↓
定期 → KPT 轻量复盘
```

## 组合 3：规划/调度 Agent

```
任务开始 → 3W（澄清目标）
    ↓
规划阶段 → SWOT + MECE
    ↓
执行中 → PDCA 小循环
    ↓
阶段完成 → AAR
    ↓
发现偏差 → 5Why
```

# 五、设计原则（给迭代 Agent 的建议）

1. **触发要明确**：失败 / 成功但不够好 / 定时触发，用不同的方法
2. **方法要适配**：不是越多越好，3-5 个核心方法够用
3. **固化要有格式**：成功经验固化成 Python 代码 + agent.md（参考 AgentFactory）
4. **验证不能少**：改进后要在类似任务上重跑，确认真的有效
5. **路由要自动**：根据场景类型自动选择最合适的方法论，而不是每次让人选
